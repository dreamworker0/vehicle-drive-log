# 차량운행일지 프로그램 종합 구조도 설계

## 목적

차량운행일지 프로그램의 사용자 관점 서비스 구조와 개발자 관점 기술 구조를 한 장에서 파악할 수 있는 가로형 구조도 이미지를 제작한다. 구조도는 실제 저장소의 코드, 설정, 문서를 근거로 하며, 구현되지 않은 구성요소는 포함하지 않는다.

## 대상 독자

- 비개발자: 사용자 역할, 주요 기능, 외부 연동 관계를 이해한다.
- 개발자·운영자: 프론트엔드 계층, Firebase 구성, Cloud Functions 유형, 데이터 흐름을 이해한다.

## 결과물

- 한글 가로형 고해상도 PNG 이미지 1개
- 한 화면에서 읽히는 계층형 아키텍처 다이어그램
- 프로젝트 안의 문서용 이미지 디렉터리에 저장
- 색상뿐 아니라 제목, 그룹 경계, 선 종류로도 의미를 구분

## 레이아웃

전체 흐름은 위에서 아래로 읽는다.

1. 사용자 역할
2. 역할별 화면과 공통 기능
3. 프론트엔드 애플리케이션 계층
4. Firebase 플랫폼과 데이터 저장소
5. Cloud Functions 서버 처리
6. 외부 서비스와 운영·품질 체계

가로 폭을 활용하여 같은 계층의 구성요소를 나란히 배치한다. 사용자 요청은 아래 방향의 실선, 서버 이벤트와 예약 작업은 점선, 외부 서비스 통신은 화살표가 있는 연결선으로 표시한다.

## 구성요소

### 1. 사용자 역할

- 시스템 관리자
- 기관관리자
- 기관직원

### 2. 화면과 핵심 기능

- 시스템 관리자: 서비스 대시보드, 기관 신청·기관 관리, 의견 관리, API 상태
- 기관관리자: 직원·차량 관리, 운행일지 관리, 예약, 통계·월간보고, 정비·주유·하이패스, 설정
- 기관직원: 오늘 현황, 운행일지 작성·OCR, 빠른 운행, 예약, 내 기록, 즐겨찾기, 차량 이력
- 공통: 로그인·기관 가입, 알림, AI 도움말, PWA 설치·오프라인 지원

### 3. 프론트엔드

- Vite 7 + React 19 + TypeScript PWA
- React Router 기반 역할별 라우팅과 AuthGuard
- 역할별·공통 React 컴포넌트
- 커스텀 Hooks 비즈니스 로직
- Firestore 도메인 모듈과 Zod 스키마
- Zustand UI 상태
- IndexedDB 기반 Firestore 오프라인 캐시와 동기화 큐
- PDF·Excel 내보내기, TMAP·OCR 클라이언트 모듈
- Service Worker, FCM, Sentry·Web Vitals

### 4. Firebase 플랫폼

- Firebase Hosting
- Firebase Authentication: Google 로그인, 역할·기관 Custom Claims
- Cloud Firestore: 멀티테넌트 데이터와 실시간 구독
- Cloud Storage: 기관 증빙서류와 의견 첨부 이미지
- Firebase Cloud Messaging: 푸시 알림
- Firestore Rules·Storage Rules: 기관 격리와 역할 기반 접근 제어

주요 데이터 도메인은 기관, 사용자, 차량, 운행일지, 예약, 주유, 하이패스, 정비, 알림, 즐겨찾기, 의견, 집계 통계다.

### 5. Cloud Functions

- 호출형: OCR, 안전한 예약 생성, 기관 가입·승인, 사용자 관리, 알림, 통계 갱신, Slack 연결, 캘린더 온디맨드 동기화
- HTTP: TMAP·공휴일 프록시, 기관 신청, 공개 의견, Slack 이벤트·OAuth, API 상태 점검
- Firestore 트리거: 문서 자동 검증, 예약·캘린더 동기화, Custom Claims, 운행거리 동기화, 의견 AI 초안, Slack 작업 처리
- 예약·배치: 예약 알림, 캘린더 역동기화, 일간 집계·백업·아카이빙, 월간 공휴일·주행거리 검증, 미활성 기관 알림

### 6. 외부 서비스

- Gemini AI: 계기판·기관 증빙 OCR, 도움말·의견 답변 초안
- Google Calendar: 차량 예약 양방향 동기화
- TMAP: POI 검색, 경로·거리·시간·통행료
- Slack: 자연어 예약 조회·생성, OAuth 연결
- 공공데이터포털: 공휴일 정보
- FCM·알림톡·이메일: 사용자·기관 알림
- Sentry: 프론트엔드와 Functions 오류 관측

## 주요 데이터 흐름

1. 사용자가 역할별 화면에서 기능을 실행한다.
2. 화면은 Hooks와 도메인 모듈을 통해 Firestore를 읽고 쓴다.
3. 권한이 필요하거나 비밀키를 사용하는 작업은 Cloud Functions를 호출한다.
4. Firestore 변경은 예약, 권한, 주행거리, 문서 검증 등의 서버 트리거를 실행한다.
5. 예약 작업은 알림, 집계, 백업, 아카이빙, 캘린더 동기화를 수행한다.
6. Cloud Functions가 외부 API와 통신하고 결과를 Firestore·FCM·사용자 화면으로 돌려준다.
7. PWA와 오프라인 캐시는 네트워크 단절 시 읽기·쓰기 흐름을 보조하고 연결 복구 후 동기화한다.

## 시각 디자인

- 밝은 중립 배경과 높은 명암 대비
- 계층별로 절제된 색상 사용: 사용자·화면, 프론트엔드, Firebase, 서버 처리, 외부 서비스
- 둥근 사각형 노드, 균일한 간격, 짧고 명확한 한글 라벨
- 장식보다 정보 전달을 우선하며 로고와 워터마크는 사용하지 않음
- 범례에는 실선, 점선, 외부 통신선의 의미를 표시

## 정확성·오류 처리

- 긴 파일명이나 개별 함수 전체 목록 대신 역할별 묶음을 사용해 과밀도를 방지한다.
- 이미지 생성 과정에서 한글이 틀리거나 연결선이 잘못된 경우 해당 부분을 수정하여 재생성한다.
- 구조도에 표시한 계층과 연결은 `README.md`, `src/App.tsx`, 역할별 Layout, `src/lib/firebase.ts`, `src/lib/firestore`, `functions/src/index.ts`, `firebase.json`, 보안 규칙을 기준으로 검증한다.
- 현재 코드와 문서가 불일치할 경우 코드와 배포 설정을 우선한다.

## 검증 기준

- 세 사용자 역할과 각 역할의 대표 기능이 모두 표시된다.
- 프론트엔드, Firebase, Cloud Functions, 외부 서비스의 경계가 명확하다.
- 사용자 요청, 데이터 접근, 함수 호출, 자동 트리거, 외부 연동 흐름이 구분된다.
- 주요 한글 라벨이 정확하고 작은 화면에서도 확대해 읽을 수 있다.
- 구현되지 않은 서비스나 데이터 흐름이 추가되지 않는다.
- PNG를 열어 잘림, 겹침, 깨진 문자, 잘못된 화살표가 없는지 확인한다.
