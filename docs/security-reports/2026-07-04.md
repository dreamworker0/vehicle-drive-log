## 이번 감사의 지배적 테마 — 멀티테넌트 **쓰기** 격리의 구조적 구멍

읽기(read) 격리는 견고하다(`belongsToMyOrg()`/`isOrgMember()`가 모든 tenant-scoped 컬렉션에 적용). 그러나 **문서가 지정한 식별자(`organizationId`·`vehicleId`·`userId`)를 규칙·트리거·콜러블이 "실제로 호출자 기관 소속인지" 검증하지 않고 신뢰**하는 패턴이 반복된다. 본문 발견 N1·N2·N3·N4는 모두 이 한 뿌리에서 나온다. Admin SDK(트리거·콜러블)는 Rules를 우회하므로, 이때 각 핸들러가 유일한 가드인데 그 가드가 빠져 있다.

## 본문 발견 (게이트 통과)

### N1 — [HIGH] orgAdmin의 자기 `organizationId` 변경 → 교차 테넌트 관리자 권한상승 ⭐

- **심각도**: HIGH (조건 충족 시 CRITICAL) · **확신도**: 9/10 · **상태**: VERIFIED(규칙+트리거 체인 추적, 독립 2회 확인) · **카테고리**: 멀티테넌트(A01)
- **위치**: [firestore.rules:66-68](../../firestore.rules#L66-L68)
  ```
  // orgAdmin은 소속 기관 사용자만 수정 가능하되, superAdmin 역할 부여 불가
  || (isOrgAdmin(resource.data.organizationId)
      && request.resource.data.role != 'superAdmin');
  ```
  `isOrgAdmin(resource.data.organizationId)`는 대상 문서의 **현재(기존)** org만 검사한다. **새 값(`request.resource.data.organizationId`)에 대한 제약이 전혀 없고**, 자기 문서 편집을 배제하지도 않는다. 그리고 [setCustomClaims.ts:29-33, 46](../../functions/src/handlers/triggers/setCustomClaims.ts#L29-L46)이 이 문서의 `role`/`organizationId`를 그대로 Auth Custom Claims로 미러링한다.
  ```ts
  const newClaims = {
      role: VALID_ROLES.includes(after.role) ? after.role : "employee",
      orgId: after.organizationId || null,
  };
  await getAuth().setCustomUserClaims(uid, newClaims);
  ```
- **익스플로잇 시나리오**:
  1. 공격자는 임의 기관 A의 admin이다(정상 가입 경로로 획득 — [joinOrganization.ts:119](../../functions/src/handlers/callable/joinOrganization.ts#L119): 기관 최초 가입자가 admin이 됨).
  2. 브라우저 콘솔에서 자기 users 문서를 수정: `updateDoc(doc(db,'users',myUid), { organizationId: '<피해 기관 B의 문서 ID>' })`. role은 `admin` 그대로 둔다.
  3. Rules 평가: isOwner 분기는 org 변경으로 fail, superAdmin 아님 → 그러나 **세 번째 분기** `isOrgAdmin(resource.data.organizationId='A')`가 true(내 현재 org=A, 내 role=admin) + `role != 'superAdmin'` → **허용**.
  4. setCustomClaims 트리거 발동 → 내 클레임이 `{role:'admin', orgId:'B'}`로 갱신.
  5. 토큰 갱신(`getIdToken(true)`) 후, `belongsToMyOrg()`/`isOrgAdmin('B')`가 전부 통과 → **기관 B의 전체 데이터를 admin 권한으로 읽기/쓰기/삭제**(차량·운행일지·예약·비용·정비·멤버 PII).
- **현실성(→ 심각도 상향 근거)**: 기관 문서 ID는 무작위지만 **쉽게 노출**된다 — [joinOrganization.ts:160-165](../../functions/src/handlers/callable/joinOrganization.ts#L160-L165)가 초대 코드 입력자에게 `orgId`를 그대로 반환하고, 전(前) 소속 직원은 자기 기관 ID를 안다(사회복지 분야 이직 빈번). 대상 ID를 알면 완전한 교차 테넌트 장악.
- **변종**: 같은 분기로 admin은 **타 소속원**의 `organizationId`도 임의 org로 바꿀 수 있다(피해자 강제 이동/추출). 근원 동일.
- **테스트 공백**: [tests/firestore-rules.test.ts:94-97](../../tests/firestore-rules.test.ts#L94-L97)는 일반 멤버의 자기 승격만 막고, **admin의 org 변경 시나리오는 커버하지 않는다.**

- **영향 한계**: 산출물은 `aiDraft`(저장)뿐이고 발송은 superAdmin 검토 후([sendFeedbackReply.ts:23](../../functions/src/handlers/callable/sendFeedbackReply.ts#L23) `requireSuperAdmin`) → 품질 오염에 그침.

## SAFE 재확인 (공격면별)

- **읽기(read) 격리 (Phase B)**: 모든 tenant-scoped 컬렉션 read가 `belongsToMyOrg()`/`isOrgMember()`로 서버단 강제. 콜러블 16개 감사 결과 superAdmin 전용 함수(sendBulkReminder·backfill*·recalculate*·refresh*·sendManual*)는 전부 `requireSuperAdmin`, admin/교차테넌트 함수(disableUser·restoreUser·withdrawOrganization·cleanupDuplicateLogs)는 orgId 클레임 대조 정상.
- **Secrets (Phase A)**: `.env`/`.env.local`/`functions/.env` 추적·히스토리 커밋 없음(.gitignore 커버). 하드코딩 실시크릿 없음(private key 블록·Discord 웹훅·AWS/Slack/GitHub 토큰 부재). 실시크릿은 전부 `defineString` 주입. CI 로그 시크릿 노출 없음, 서비스계정 mktemp→rm.
- **스킬 공급망 (Phase D)**: [sync-claude-agents.ts](../../scripts/sync-claude-agents.ts)는 readdir 엔트리명만 사용(경로 traversal 불가) + 마커 기반 `.claude/` 내부 삭제. SKILL.md/워크플로우에 데이터 탈취·임의 명령·무승인 배포 유도 없음(오히려 시크릿 하드코딩 금지 지침 강화).
- **SSRF (Phase B)**: tmapProxy·holidayProxy 모두 업스트림 호스트 하드코딩(`apis.openapi.sk.com`/`apis.data.go.kr`), 사용자는 쿼리 값만 제어 → 오픈 프록시 아님. 인증+IP 레이트리밋 강제.
- **자기 권한상승 (Phase B)**: 일반 멤버의 자기 role 변경은 [firestore.rules:56-57](../../firestore.rules#L56-L57)이 차단(테스트 커버). setCustomClaims는 잡값 role을 employee로 강등.
- **의존성 (Phase 3 위임)**: `npm run audit`(root+functions) → **취약점 없음**.
