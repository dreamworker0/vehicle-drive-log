# 개선 계획서 — 구글 캘린더 → 앱 예약 동기화 UX (2026-07-12)

> **배경(왜 만드는가)**: 사용자가 구글 캘린더(차량 전용 캘린더)에 `7/13 11:30 합정역` 일정을
> 직접 등록했으나 앱 예약에 반영되지 않는다고 보고. 조사 결과 **동기화 기능 자체는 정상**이며,
> "당겨오기(pull)" 트리거가 걸리는 시점의 **구조적 사각지대**가 원인이었다. 사용자가 캘린더에서
> 직접 등록하는 흐름을 앱이 신뢰성 있게·투명하게 반영하도록 개선한다.
>
> **범위(2026-07-12 사용자 확정)**: 실시간 Webhook은 제외한 **빠른 개선** + 수동 등록 일정 품질 개선.
> 실시간 Webhook은 유지보수 부담이 주 7시간·1인 운영 규모에 과설계라 채택하지 않음(§4 참조).
>
> **가용 시간**: 주 ≈7시간(평일 1h 조각 / 주말 2h 묶음).

## 0. 진단 — 현재 역방향(캘린더 → 앱) 동기화 동작

차량(기관) 캘린더 ↔ 예약은 서비스 계정(ADC) 기반 **폴링 방식**으로 양방향 동기화된다. 캘린더에서
직접 만든 이벤트가 앱 예약으로 당겨지는 경로는 **딱 두 가지**뿐이다.

| 경로 | 코드 | 언제 도는가 | 이번 사례에서 |
|---|---|---|---|
| **스케줄 자동** `syncCalendarToApp` | `functions/src/handlers/scheduled/calendarSchedule.ts:33` | cron `0 6-22 * * 1-5` + 코드 내부 주말/심야 방어 스킵(:47) → **평일만** | 오늘 일요일 → **실행 안 됨**. 다음 실행 월 06:00 |
| **온디맨드** `triggerOnDemandCalendarSync` | `functions/src/handlers/callable/triggerOnDemandCalendarSync.ts` | **예약 캘린더 화면 진입 시에만** 자동 호출(`src/hooks/reservationCalendar/useReservationData.ts:97`), 차량별 30분 쿨다운 | 사용자는 "오늘의 운행"(홈) 화면만 봄 → **트리거 안 됨** |

**결론**: 버그가 아니라 (1) 스케줄러 주말 공백 + (2) 온디맨드 트리거가 예약 캘린더 화면에만 걸림 +
(3) 수동 강제 수단·동기화 상태 가시성 부재가 겹친 **사각지대**. 홈의 "내 예약"은 *오늘* 예약만
표시하므로 내일(7/13) 예약은 애초에 안 보임.

### 부차 이슈(동기화가 돌아도 남는 것)
- 수동 등록 제목 `합정역`은 앱 표준 형식 `[차량명] 목적지 — 예약자`가 아님. 파서
  (`functions/src/services/calendar/calendarSync.ts:182` `parseEventToReservation`)는 예약을
  생성하되 `destination=합정역`만 채우고, **예약자는 이벤트 생성자(creator) 이메일이 앱 계정과
  일치할 때만** 채워짐(`calendarSchedule.ts:302` `findUserByEmail`). 용도는 빈 값.
- 이벤트가 **해당 차량에 연동된 캘린더**(`vehicles.googleCalendarId`)에 있어야 함.

## 1. 개선 항목

### 항목 A — 수동 "지금 동기화" 버튼 + 마지막 동기화 시각 표시 (예약 캘린더 화면)
사용자가 스스로 즉시 당겨오고, 언제 동기화됐는지 눈으로 확인할 수 있게 한다. **가장 체감 효과가 큰
저위험 개선.**

- **대상**
  - `src/hooks/useCalendarSync.ts` — 수동 호출용 경로 추가. 기존 `syncVehicleOnDemand`에
    `force` 옵션을 받아 **수동 클릭 시 30분 쿨다운을 우회**(`checkCooldown` 스킵)하도록 확장.
    `getSyncTimeMap()`을 노출해 마지막 동기화 시각을 UI가 읽게 함.
  - `src/hooks/useReservationCalendar.ts` / `src/hooks/reservationCalendar/useReservationData.ts`
    — 수동 트리거 함수(`syncNow`)와 마지막 동기화 시각을 반환값에 추가. 완료 후 `fetchReservations()` 재호출.
  - `src/components/common/ReservationCalendar.tsx` — 캘린더 헤더에 `🔄 지금 동기화` 버튼 +
    `마지막 동기화 HH:MM`(없으면 "동기화 이력 없음") 표시. 로딩 중 스피너.
- **재사용**: `triggerOnDemandCalendarSync` 콜러블·`useCalendarSync`의 재시도/백오프 로직 그대로.
  토스트는 `useToast().showToast`.
- **완료 기준**: 버튼 클릭 → 즉시 동기화 → 예약 목록 리프레시 → 마지막 시각 갱신. 쿨다운 중이어도
  수동 버튼은 동작. 실패(공유 끊김) 시 기존 `calendar-not-found` 메시지 토스트.

### 항목 B — 온디맨드 동기화를 홈(오늘의 운행) 진입 시에도 트리거
예약 캘린더를 열지 않아도 앱을 켜면 최신 캘린더 상태가 반영되도록 확대한다.

- **대상**: `src/components/employee/TodayDashboard.tsx` — 마운트 시 `useCalendarSync`로 조직 연동
  차량을 백그라운드 동기화. **30분 쿨다운은 유지**(남용·비용 방어).
- **비용 주의**: 홈은 접근 빈도가 높다. 차량 수만큼 콜러블이 호출되므로 **반드시 기존 30분 쿨다운을
  유지**하고, `googleCalendarId`가 유효한 차량으로만 한정한다(`useReservationData.ts:102-105` 패턴 재사용).
  → 참고: `.agent/skills/firebase-cost-reduction`.
- **완료 기준**: 홈 진입 시 쿨다운을 지난 연동 차량만 백그라운드 동기화. 콘솔/비용 급증 없음.

### 항목 C — 수동 등록 일정의 예약 품질 개선
캘린더에서 자유형식(`11:30 합정역`)으로 등록해도 예약자/목적지가 최대한 채워지게 하고, 사용자가
권장 형식을 알도록 안내한다.

- **C-1 파서/매칭 보강** — `functions/src/services/calendar/calendarSync.ts` `parseEventToReservation`
  및 `calendarSchedule.ts` 생성 분기(:296-313):
  - 예약자(`reservedByName`)가 비고 creator `displayName`도 없을 때, creator 이메일 로컬파트 등
    최소한의 폴백을 채워 "예약자 미상"으로 남지 않게 한다(개인정보 노출은 이메일 전체 저장 금지 —
    현행처럼 `creatorEmail`은 Firestore 미저장 유지).
  - 제목에 구분자 없이 목적지만 있는 케이스(현재 `destination`만 채움)는 그대로 두되, 회귀 방지
    단위 테스트로 고정.
- **C-2 안내 강화** — `src/components/admin/VehicleCalendarSection.tsx` "📋 설정 방법 안내"
  (`:186` 이하 `<ol>`)에 항목 추가:
  - "직접 일정 등록 시 제목을 `[차량명] 목적지 — 예약자` 형식으로 작성하면 예약자·목적지가 정확히
    반영됩니다. 설명란에 `예약자:`, `용도:`, `목적지:` 줄을 넣어도 됩니다."
- **완료 기준**: `11:30 합정역`(생성자=앱 사용자)으로 등록 시 예약자가 채워짐. 안내문에 권장 형식 노출.
  파서 회귀 테스트(`functions/src/__tests__/`) 통과.

## 4. 하지 않기로 한 것 (근거)

| 항목 | 이유 |
|---|---|
| **실시간 Webhook(Google Calendar `watch`)** | 채널 등록 + ~7일마다 채널 갱신 스케줄러 + 수신 HTTP 엔드포인트 + `syncToken` 증분 동기화까지 상시 유지해야 함. 주 7h·1인 운영 규모에 유지보수 부담이 과함. 폴링(평일 매시 + 주말 대비 온디맨드/수동 버튼)으로 체감 지연이 해소되면 충분. 실제로 "즉시성"이 반복 요구될 때 재검토. |
| **주말 스케줄 동기화 추가** | 이번 범위(빠른 개선)에서 제외. 항목 A(수동 버튼) + 항목 B(홈 트리거)로 주말에도 사용자가 앱을 열면 반영되므로 스케줄러를 굳이 주말에 돌려 함수 호출 비용을 늘리지 않음. 항목 A/B 관찰 후에도 주말 공백이 문제로 남으면 그때 저빈도 추가. |
