## 아키텍처는 모듈화되어 있으나 일부 복잡도 집중이 남아 있다

프런트엔드는 컴포넌트, 훅, Zustand 스토어, Zod 스키마, 도메인별 Firestore 모듈로 구분되어 있다. Functions도 callable, HTTPS, scheduled, trigger 핸들러와 서비스·코어·유틸리티 계층으로 나뉜다. `index.ts`가 초기화와 export 역할만 담당하도록 제한한 점도 적절하다.

TypeScript의 `strict` 모드가 프런트엔드와 Functions에 적용되고, 프런트엔드에서는 `no-explicit-any`가 오류로 설정되어 있다. 또한 조직 범위 데이터 쿼리에 `organizationId` 필터를 요구하는 프로젝트 전용 ESLint 규칙이 있어 멀티테넌트 실수를 정적 분석 단계에서 차단한다.

반면 다음 파일은 400줄을 넘으며 여러 책임이 집중되어 있다.

- `functions/src/services/statistics/dashboardSections.ts`: 559줄

- `functions/src/handlers/triggers/reservationTriggers.ts`: 417줄

프로덕션 코드에도 일부 `any`, `@ts-expect-error`, 훅 의존성 규칙 예외가 남아 있다. 현재 린트는 통과하지만, 기능 확장 시 변경 영향 범위가 커질 수 있으므로 대형 파일은 계산·입출력·렌더링 책임을 기준으로 추가 분리하는 것이 좋다.

## 테스트 수는 충분하지만 전체 단위 커버리지는 낮다

프런트엔드 테스트 실행 결과는 모두 통과했지만 전체 커버리지는 다음과 같다.

| 측정 지표 | 실제 커버리지 | 현재 게이트 |
|---|---:|---:|
| Statements | 27.90% | 27% |
| Branches | 17.83% | 17% |
| Functions | 21.87% | 21% |
| Lines | 29.13% | 28% |

게이트가 현재 실측치보다 약 1%p 낮게 설정되어 있어 회귀 방지선 역할은 하지만, 많은 관리자·직원 화면, PDF 생성, Tmap 연동 코드가 0% 또는 낮은 커버리지로 남는다. 테스트 개수는 많지만 순수 계산 함수와 일부 훅에 집중되어 있으며, 전체 화면과 오류 처리 경로의 단위 테스트 깊이는 상대적으로 얕다.

다만 일반 E2E와 인증 에뮬레이터 E2E가 이를 일부 보완한다. 실제 로그인 역할 분기, 직원 운행일지 작성, 관리자 Excel 내보내기, 예약 승인·반려, 오프라인 작성 후 온라인 동기화까지 통과했다. 따라서 현재 테스트 체계는 “넓은 통합 검증과 제한적인 단위 커버리지” 형태로 판단한다.

Functions는 251개 단위 테스트가 통과했으나 커버리지 임계치를 CI에서 별도로 강제하지 않는다. 비용이 큰 OCR·AI 경로, 조직 삭제·복원, 스케줄 배치의 예외 경로부터 Functions 커버리지 게이트를 추가하는 것이 적절하다.

## 데이터 격리는 강하지만 API 남용 방어는 보강이 필요하다

Firestore Rules는 Custom Claims의 `orgId`를 기준으로 차량, 운행일지, 예약, 주유, 하이패스, 정비 데이터 읽기를 기관 단위로 제한한다. 주요 문서의 `organizationId` 변경을 금지하고 OAuth 비밀 서브컬렉션은 클라이언트 읽기·쓰기를 모두 차단한다. Storage Rules도 사용자 폴더, 파일 크기, 이미지 MIME 유형을 제한한다. Rules 에뮬레이터 테스트 12개가 모두 통과해 이 설계가 자동으로 검증되고 있다.

Functions의 핵심 API 역시 Firebase Auth, 역할, 조직 소속을 재검증하고, 외부 API 프록시는 허용된 origin, 인증 토큰, IP 기반 Rate Limit을 공통 팩토리에서 적용한다. `.env`와 `functions/.env`는 Git에서 제외되어 있으며, 의존성 보안 감사에서도 알려진 취약점이 발견되지 않았다.

주요 위험은 다음과 같다.

1. 코드 검색 기준 19개 위치에서 `enforceAppCheck: false`가 확인된다. 브라우저 App Check 초기화 코드가 있어도 Functions가 토큰을 강제하지 않으면 자동화된 남용 방어 효과가 제한된다.
2. UID/IP Rate Limit은 Firestore 접근 실패 시 요청을 허용하는 fail-open 방식이다. 일반 기능 가용성에는 유리하지만 OCR·AI·공개 접수처럼 비용 또는 스팸 위험이 있는 API에서는 장애 시 남용 가능성이 커진다.
3. 기관 신청 API는 업로드 크기를 제한하지만 전달받은 MIME 유형의 명시적 화이트리스트 검증을 추가할 여지가 있다.
4. Hosting CSP에 `unsafe-inline`이 포함되어 있어 외부 스크립트 호환성은 높지만 XSS 방어 심도는 낮아진다.

## PWA와 코드 분할은 우수하지만 번들 예산 여유가 작다

비로그인 사용자를 위한 경량 진입점과 인증 사용자의 전체 진입점이 분리되어 있고, Recharts, Leaflet, XLSX, Sentry, Firebase 모듈은 수동 청크 또는 지연 로딩을 사용한다. 서비스워커에는 정적 자산 프리캐시, Google Fonts·Storage·지도 타일의 만료 정책, 백그라운드 동기화, 알림 클릭 처리가 구현되어 있다.

빌드 결과는 모든 예산을 통과했지만 여유가 크지 않다.

| 번들 지표 | 실제 | 예산 | 사용률 |
|---|---:|---:|---:|
| 전체 JavaScript 원시 크기 | 2,916.4KB | 3,000KB | 97.2% |
| 전체 JavaScript gzip | 917.0KB | 950KB | 96.5% |
| 최대 JavaScript 청크 | 398.9KB | 420KB | 95.0% |
| 전체 CSS 원시 크기 | 182.3KB | 200KB | 91.2% |
| 전체 CSS gzip | 28.2KB | 35KB | 80.6% |

서비스워커 자체도 원시 494.4KB, gzip 153.08KB다. 전체 수치는 모든 청크의 합계이므로 초기 다운로드 크기와 같지는 않지만, 향후 기능 추가 시 현재 예산을 빠르게 초과할 가능성이 높다. Recharts, Firebase DB, XLSX가 가장 큰 청크이므로 기능별 로딩 시점과 실제 사용률을 우선 점검해야 한다.

평가 환경의 Node.js는 24.14.0이었고 프로젝트 권장 버전은 22.x였다. 빌드는 통과했으나 현재 사전 검사는 버전 불일치를 경고만 하고 실패시키지 않는다. CI는 Node 22를 고정하므로 배포 재현성은 유지되지만, 로컬 개발에서는 `--strict` 사용 또는 버전 관리자 자동 적용을 권장한다.

## 운영 자동화와 관측성은 상용 서비스 수준이다

CI는 린트, 양쪽 타입 검사, 빌드, 커버리지 게이트, Functions 단위 테스트, Firestore Rules, 일반 E2E, 인증 E2E, 번들 예산, 보안 감사를 순서대로 수행한다. 배포 워크플로는 CI가 검증한 커밋 SHA를 배포하고 Functions·Rules를 Hosting보다 먼저 반영하며 동시 배포를 방지한다.

프런트엔드는 Sentry를 동적으로 로딩해 비로그인 초기 번들 비용을 줄이고, 인증 사용자·역할·기관 태그와 브라우저 라우팅 트레이스를 기록한다. 프로덕션에서는 CLS, FCP, LCP, TTFB, INP도 수집한다. Functions에도 별도 Sentry 초기화가 존재하며, API 상태 점검과 운영 배치가 마련되어 있다.

다만 `workflow_dispatch` 수동 배포는 CI를 우회할 수 있고 워크플로에 GitHub Environment 승인 단계가 명시되어 있지 않다. 긴급 배포 경로를 유지하더라도 운영자 승인 또는 최소 검증 세트를 강제하는 것이 안전하다. Firebase CLI도 프로젝트 의존성에 명시적으로 고정해 CI와 로컬의 재현성을 높일 필요가 있다.

## 문서는 풍부하지만 실제 코드와 일부 차이가 있다

README, 프로젝트 문서, 운영 매뉴얼, 기여 가이드, 롤백 문서, 변경 이력이 갖춰져 있어 신규 개발자와 운영자가 시스템을 이해하기 쉽다. 기능 목록, 역할, 배포 명령, Functions 종류, 테스트 체계까지 문서화되어 있다는 점은 높은 점수의 근거다.

다음 항목은 현재 코드와 동기화가 필요하다.

- README의 프런트엔드 테스트 49파일·357개, Functions 19 suite·172개 수치는 실제 실행 결과보다 오래되었다.
- README는 Functions를 TypeScript ESM으로 설명하지만 현재 Functions TypeScript 설정은 `module: commonjs`다.
- PROJECT 문서는 인증 클레임을 `organizationId`로 설명하지만 실제 Custom Claims와 예약 코드는 `orgId`를 사용한다.
- 기본 E2E 설명의 spec 파일 수와 제외 상태도 현재 구성과 차이가 있다.

이 차이는 실행 오류를 만들지는 않지만, 신규 기능 구현과 보안 검토 시 잘못된 계약을 참조하게 할 수 있다. CI에서 문서의 테스트 수치를 자동 생성하거나, 변동이 잦은 수치는 정확한 개수 대신 실행 명령과 최소 기준으로 표현하는 방식을 권장한다.

## 개선 우선순위

### P0 — 즉시 검토

1. **비용·데이터 변경 API에 App Check를 단계적으로 강제한다.**
   - 우선 대상: OCR, AI, 예약 생성, 관리자 알림, 조직 가입·복원·탈퇴
   - 모니터링 모드로 유효 토큰 비율을 확인한 뒤 강제 전환하는 방식이 안전하다.

2. **고위험 API의 Rate Limit 장애 정책을 구분한다.**
   - 일반 조회는 기존 fail-open을 유지할 수 있다.
   - OCR·AI·공개 접수는 짧은 로컬 제한, Cloud Armor, App Check 또는 제한적 fail-closed를 조합한다.

### P1 — 다음 품질 개선 주기

3. **프런트엔드 라인 커버리지를 40%, 브랜치를 25% 이상으로 단계 상향한다.**
   - 우선 대상: 예약 편집·취소, 운행일지 제출, PDF·Excel 생성, 관리자 권한 변경, 외부 API 실패 경로

4. **Functions 커버리지 측정과 임계치를 CI에 추가한다.**
   - 인증·권한 거부, 외부 서비스 실패, 배치 멱등성, 부분 실패 복구를 우선 검증한다.

5. **색상 대비 예외 4건을 수정하고 Axe 게이트에 포함한다.**
   - 이번 실행에서 이용약관과 개인정보 처리방침 페이지에서 각각 2건이 보고되었다.

### P2 — 지속 개선

6. **번들 예산 여유를 최소 15% 이상 확보한다.**
   - Recharts·XLSX·Firebase DB의 실제 로딩 시점과 대체 경량 빌드를 검토한다.

7. **400줄 이상 파일을 책임 단위로 분리한다.**
   - 통계 계산, Firestore 입출력, 응답 조립, React 렌더링을 분리 기준으로 삼는다.

8. **문서의 테스트 수치·모듈 형식·클레임 이름을 현재 코드와 일치시킨다.**

9. **수동 배포에 승인 또는 최소 품질 게이트를 추가하고 Firebase CLI 버전을 고정한다.**

## 평가 범위와 한계

이번 평가는 다음을 포함한다.

- 저장소 구조, 주요 소스 코드, 설정, 보안 규칙, 문서 검토
- 로컬 린트·타입 검사·프로덕션 빌드
- 프런트엔드·Functions 단위 테스트
- Firestore Rules 및 일반·인증 E2E 테스트
- 의존성 취약점 감사와 번들 크기 확인

다음 항목은 포함하지 않았다.

- 실제 프로덕션 데이터의 정확성 및 개인정보 처리 실태 감사
- 외부 전문 도구를 사용한 침투 테스트
- 동시 사용자 부하·장시간 안정성·비용 시험
- iOS·Android 실기기 전수 사용성 평가
- Google Calendar, Gemini, FCM, 알림톡 등 외부 서비스의 실환경 장애 주입 시험

따라서 84점은 “현재 저장소와 로컬 자동검증에서 확인 가능한 기술 품질”을 의미하며, 보안 인증이나 실운영 SLA를 보증하는 점수는 아니다.

## 추가로 확인할 질문

- 프로덕션에서 App Check 유효·무효 요청 비율은 어느 정도인가?
- 실제 사용자가 처음 진입할 때의 LCP·INP·CLS 분포는 기기와 네트워크별로 어떠한가?
- OCR·AI·Tmap·Calendar API의 기관별 비용 상위 구간과 실패율은 어떠한가?
- Firestore Rules 테스트가 모든 민감 컬렉션의 교차 기관 읽기·쓰기·필드 변조를 포함하는가?
- 백업 파일의 실제 복구 훈련과 RTO/RPO 검증은 최근 언제 수행되었는가?

이 질문에 대한 운영 데이터가 확보되면 보안·성능·운영 점수를 보다 정확하게 재평가할 수 있다.
