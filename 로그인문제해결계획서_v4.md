## 1. 현재 상황 및 문제점 요약

차량운행일지 서비스의 구글 로그인(Firebase Auth) 이슈는 사용자가 서비스에 아예 접근하지 못하게 만드는 치명적 문제를 야기해 왔습니다. 주요 원인:

- **엄격한 보안 정책의 역효과**: `permission-denied` 발생 시 보안을 위해 사용자를 강제 로그아웃시키는 로직이 작동했었음 (현재는 제거됨, `08dbaed`).
- **세션-토큰 불일치**: 모바일/PWA/인앱 브라우저에서 백그라운드 전환 시 토큰 만료 또는 App Check 검증 실패가 빈번.
- **무한 로그아웃 루프**: 토큰 만료 + 세션 잔존 → 403 → 강제 로그아웃 → 자동 재로그인 → 토큰 오류 → 403 반복.

## 2. 해결 방향: "운영 안정성 우선, 데이터 기반 보안 정책 재조정"

**"포기할 보안은 포기하되, 무차별 완화는 금지하고 실제 발생률을 보고 결정한다"**가 v4의 방침입니다.

### 🛑 포기/완화할 보안 정책
1. 데이터 접근 실패 시 즉각적인 세션 파기 중단 ✅ 이미 적용 (`08dbaed`)
2. App Check 등 환경 검증 실패 시 하드 블로킹 완화 → §4-0
3. **단**, 무차별 완화(`request.auth != null` 등 격리 붕괴)는 절대 금지

### 🟢 강화할 운영 정책
1. 어떤 에러가 발생해도 세션은 유지 — **단, `firebaseUser==null`(진짜 만료/삭제)은 예외** ✅ 이미 [useAuth.tsx:216-224](src/hooks/useAuth.tsx#L216-L224)에서 처리
2. 에러 발생 시 친절한 UI 알림 + **재연결 버튼**(§4-3)

## 4. 구체적인 실행 계획 (Action Plan v4)

### Step 0: App Check 강제 정책(Enforcement) 완화 — 즉시 조치 (콘솔)

가장 빠른 운영 안정화 수단. Firebase Console → App Check → Firestore/Functions의 Enforce가 켜져 있다면 **Unenforced로 전환**.
- **즉시 조치**: 콘솔 설정 확인 → Unenforced 전환
- **장기 대안**: Unenforced 방치 시 할당량 공격 위험이 있으므로, 안정화 이후 ReCaptcha Enterprise v3 Score Threshold 완화 또는 모바일/인앱 브라우저의 `initializeAppCheck` 지연 여부 점검을 후속 과제로 계획.

### Step 1: 강제 로그아웃 로직 — 잔여 경로 정리 (작업 시간 ~30분)

grep 결과 `auth.signOut()` 자동 호출 경로는 이미 제거됨. 다만 `permission-denied`를 직접 catch하는 4곳에서 세션을 건드리지 않는지 확인:

- [src/hooks/useVehicleManager.ts](src/hooks/useVehicleManager.ts)
- [src/hooks/useTodayDashboard.ts](src/hooks/useTodayDashboard.ts)
- [src/components/admin/AdminDashboard.tsx](src/components/admin/AdminDashboard.tsx)
- [src/components/superAdmin/SuperAdminLayout.tsx](src/components/superAdmin/SuperAdminLayout.tsx)

**확인 사항**:
- `permission-denied` catch 블록 내 `signOut`/`auth` 접근 코드 없는지
- Sentry로 보내는 경로 있다면 `permission-denied`만 필터링
- **대체 동작**: 상태값(`isError`, `isSyncing`)만 변경 → "데이터 동기화 지연" 스낵바/배지 표시

### Step 2: Sentry 모니터링 (1~2주) — 데이터 기반 후속 결정

Step 0/1 적용 후 **1~2주 모니터링 기간**을 갖고, 실제로 무엇이 남는지 데이터로 확인합니다. 모니터링 지표는 §5.1 참조.

이 기간에 수집할 핵심 신호:
- `permission-denied`가 여전히 발생한다면 **어떤 경로**에서 발생하는가? (useAuth 초기 / 페이지 훅 / 신규가입 직후)
- 신규가입 race condition 사례가 실제로 보고되는가?
- App Check Unenforced 전환 후 비정상 트래픽이 보이는가?

이 데이터에 따라 아래 Step 3/4/5의 적용 여부와 강도를 결정합니다.

### Step 3: 재연결 토스트 버튼 — 사용자 자가 복구 경로 (즉시 적용 가능)

`permission-denied` 2회 재시도 실패 후 [useAuth.tsx:190](src/hooks/useAuth.tsx#L190)에서 토스트만 띄우는데, **여기에 "다시 로그인" 버튼**을 추가합니다.

- 버튼 동작: `sessionStorage.clear()` + `auth.signOut()` + 로그인 화면 리다이렉트 + `window.location.reload()`
- 진짜 만료(`firebaseUser==null`)는 이미 [useAuth.tsx:216-224](src/hooks/useAuth.tsx#L216-L224)에서 자동 처리되므로 별도 `onIdTokenChanged` 핸들러는 도입하지 않습니다.

### Step 4 (조건부): 신규가입 race condition 강화

**조건**: Step 2 모니터링에서 신규가입 첫 진입 403이 주 1회 이상 보고되는 경우.

- 1차 시도: [useAuth.tsx:131](src/hooks/useAuth.tsx#L131)의 `setTimeout(startOrgWatch, 500)`을 **1500~2000ms로 확대** (1줄 변경).
- 2차 시도(여전히 발생 시): users 문서 존재를 짧은 폴링으로 확인하는 로딩/스플래시 화면 도입.

폴링 스플래시는 컴포넌트/상태/타이머가 추가되는 변경이므로, **데이터로 필요성이 입증되기 전엔 진행하지 않습니다.**

### Step 5 (조건부): Firestore 룰 Claims Fallback

**조건**: Step 2 모니터링에서 `useAuth.tsx:142-151`의 강제 갱신 로직을 통과했음에도 `permission-denied`가 지속되는 케이스가 확인되는 경우.

- 룰 수정 예시:
  ```javascript
  function userOrgId() {
    return request.auth.token.orgId != null
      ? request.auth.token.orgId
      : get(/databases/$(database)/documents/users/$(request.auth.uid)).data.organizationId;
  }
  ```
- **비용 주의**: 룰 내 `get()` 호출은 매 read마다 1회 추가 과금. 리스트 쿼리에는 적용되지 않으므로 효과가 제한적입니다. **반드시 클라이언트 강제 갱신 로직이 실패하는 케이스가 데이터로 확인된 후에만** 진행하세요.

## 5. 성공 지표 및 롤백 기준

### 5.1 성공 지표 (배포 후 1~2주 모니터링)
- Sentry `permission-denied` 발생률: 기준 대비 **80% 이상 감소**
- 일일 강제 로그아웃 이벤트(`auth/sign-out` 자동 트리거) 수: **0건 수렴**
- 사용자 문의(로그인/접속 불가) 티켓 수: 기준 대비 **50% 이상 감소**
- **신규 측정 지표**: 신규가입 첫 진입 시 `permission-denied` 발생 건수 (Step 4 결정 근거)

### 5.2 롤백 기준
- Step 0 (App Check Unenforced): 비정상 트래픽/할당량 폭증 시 즉시 Enforced 복귀
- Step 5 (룰 fallback): 의도치 않은 타 기관 데이터 노출이 1건이라도 확인되면 **즉시 롤백**
- Firestore 읽기 비용(Billing) 비정상 상승 시 Step 5 재검토

## 7. 작업 순서 요약 (v4)

1. **(0순위, 즉시)** App Check Enforcement 콘솔 점검 → Unenforced 전환
2. **(1순위, ~30분)** Step 1 — 4개 훅/컴포넌트 일관성 점검
3. **(1순위)** Step 3 — 재연결 토스트 버튼 추가
4. **1~2주 Sentry 지표 / Billing 모니터링** (Step 2)
5. **데이터 기반 후속 결정**:
   - 신규가입 race가 보고되면 Step 4 (지연 확대 → 필요 시 폴링)
   - useAuth 강제 갱신 후에도 `permission-denied`가 지속되면 Step 5 (룰 fallback)
6. 안정화 후 App Check Score Threshold 재조정 등 보안 회복 작업

## 8. v3 → v4 핵심 변경 사유 요약

- **중복 제거**: v3의 Step 2/3 중 일부는 현재 코드에 이미 구현되어 있어 새 작업으로 분류하면 안 됨.
- **ROI 우선**: 룰 변경(Step 5)과 폴링 스플래시(Step 4 2차)는 **변경 비용 대비 효과가 불확실**하므로 데이터로 입증 후 진행.
- **소규모 서비스에 맞는 단계화**: 한 번에 모든 방어막을 쌓기보다 "측정 → 결정" 사이클을 한 번 끼워 넣어 운영 복잡도와 비용 누수를 차단.
