# 기획·스코프 리뷰 (Planning & Scope Review)

> 비자명한 새 기능·큰 변경을 **구현하기 전에** 거치는 사전 기획 절차.
> Claude Code plan mode 진입 시, 또는 요청이 모호·광범위할 때 적용한다.
> 버그픽스·자잘한 수정엔 적용하지 않는다(아래 Hold 모드의 경량 버전으로 충분).
>
> gstack `/ceo`(plan-ceo-review)의 사전 기획 원칙을 이 프로젝트에 맞춰 흡수한 것이다.
> plan mode·[Plan 에이전트]와 중복되는 "계획→승인" 골격은 그쪽에 맡기고, 여기서는
> plan mode가 **강제하지 않는 델타**(전제 도전·대안 제시·가드레일·스코프 모드)만 규정한다.

## 핵심 원칙

구현 전에 세 가지를 합의한다: **(1) 올바른 문제를 푸는가, (2) 최소안과 이상안, (3) 스코프.**
사용자가 100% 통제한다 — 모든 스코프 변경은 명시적 opt-in이며, 몰래 추가·삭제하지 않는다([agents.md D19·§5.4](../agents.md) 재확인).

## Step 1 — 전제 도전

- 이게 진짜 문제인가, 아니면 대리 지표(proxy)인가?
- 실제 사용자 결과(outcome)는 무엇인가?
- **아무것도 안 하면 뭐가 깨지나?** (깨지는 게 없으면 스코프 후보)

## Step 2 — 기존 코드 레버리지

- 하위 문제를 **기존 코드·파일에 매핑**한다. 신규 작성 전에 "기존 파일 편집으로 끝나는지" 먼저 확인(CLAUDE.md 작업 스타일).
- 리팩토링으로 해결 안 될 때만 새로 만든다.

## Step 3 — 구현 대안 (필수)

- 접근법 **2~3개**를 제시한다. 반드시 **최소안 1개 + 이상안 1개**를 포함.
- 각각 노력·리스크·장단점을 1~2줄로. 사용자가 하나를 고른 뒤 진행한다(임의 선택 금지).

## Step 4 — 스코프 모드

| 모드 | 자세 | 이 프로젝트 기본값 |
|---|---|---|
| **HOLD** | 명시 스코프에 최대 엄격(실패 모드·엣지케이스·관측성). 스코프 변경 없음 | 버그픽스·핫픽스·리팩토링 |
| **SELECTIVE** | 현재 스코프를 baseline으로 굳히고, 확장 기회는 **개별 opt-in**으로 따로 제안 | **새 기능·기존 기능 개선의 기본** |
| **REDUCTION** | 핵심 결과를 내는 최소 버전으로 외과적 축소. 나머지는 후속으로 분리 | 과잉·불명확하다고 느껴질 때 |
| **EXPANSION** | 10배 가치를 적극 발굴(10-star) | **사용자가 명시적으로 "크게 가자"고 할 때만** |

- **EXPANSION은 기본값이 아니다.** 이 프로젝트는 규모에 맞는 축소 편향이 원칙이다(CLAUDE.md 작업 스타일 — 명시한 변경만, 자잘한 끼워넣기 금지). 확장은 사용자가 먼저 요청해야 가동한다.
- 모드를 정한 뒤에는 **그 모드에 commit**한다 — 나중에 슬그머니 스코프를 늘리거나 줄이지 않는다.

## Step 5 — 가드레일 (스코프의 일부로 취급)

- **명명된 에러**: "에러 처리"라고 뭉뚱그리지 말고 — 예외·트리거·캐처·사용자 영향을 명시.
- **섀도 경로**: 모든 데이터 흐름에 대해 nil/빈/상류 에러 입력을 추적.
- **엣지케이스**: 더블클릭, 작업 중 이탈, 느린 연결, stale 상태, 뒤로가기.
- **관측성도 스코프**: Sentry 로깅·토스트·health 체크를 일급 산출물로.
- **미룬 작업은 기록**: [docs/차량운행일지_구현계획서.md](../../docs/차량운행일지_구현계획서.md) 또는 TODO에 적지 않으면 존재하지 않는 것.

## 산출 & 마무리

- 계획은 plan mode의 ExitPlanMode로 **사용자 승인**을 받고 구현에 들어간다.
- 시간 추정 시 사람-시간과 CC 압축 시간을 함께 제시할 수 있다(선택).
- 작업 완료 후 구현계획서에 Phase로 기록(프로젝트 관행).
