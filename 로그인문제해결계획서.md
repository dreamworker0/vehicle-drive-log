# 구글 로그인 문제 해결 및 운영 안정화 계획서

## 1. 현재 상황 및 문제점 요약

최근 차량운행일지 서비스에서 지속적으로 발생하고 있는 구글 로그인(Firebase Auth) 관련 이슈들은 사용자가 서비스에 아예 접근하지 못하게 만드는 치명적인 문제를 야기하고 있습니다. 주요 원인은 다음과 같습니다.

*   **엄격한 보안 정책의 역효과**: Firestore 데이터 접근 시 권한 오류(Permission Denied, 403)가 발생하면 보안을 위해 사용자를 **강제로 로그아웃**시키는 로직이 작동했습니다.
*   **세션과 토큰의 불일치**: 모바일 브라우저나 PWA 환경(특히 인앱 브라우저 등)에서 백그라운드 전환 시 토큰이 만료되거나 Firebase App Check 검증에 실패하는 경우가 빈번합니다.
*   **무한 로그아웃 루프**: 토큰은 만료되었으나 브라우저 세션은 남아있는 상태에서, 데이터를 요청하면 403 오류가 발생하고 -> 로그아웃 처리 -> 다시 자동 로그인 시도 -> 토큰 오류 -> 403 발생이 반복되는 **무한 루프**에 빠졌습니다.

## 2. 해결 방향: 보안 타협을 통한 '운영 안정성' 확보

**"이제는 보안 문제를 포기할 것은 포기하고 운영의 방향으로 진행한다"**는 방침에 따라, 시스템의 우선순위를 재조정합니다.
사용자가 데이터를 조금 늦게 보거나 일부 검증이 약화되더라도, **서비스 접속 자체가 차단되는 일은 없어야 합니다.**

### 🛑 포기/완화할 보안 정책
1.  **데이터 접근 실패 시 즉각적인 세션 파기 (강제 로그아웃) 중단**
2.  **지나치게 엄격한 Firestore 보안 규칙 완화** (필요시 읽기 권한의 검증 문턱을 낮춤)
3.  **App Check 등의 환경 검증 실패 시 하드 블로킹(Hard Blocking) 완화**

### 🟢 강화할 운영 정책
1.  **어떤 에러가 발생해도 세션(로그인 상태)은 유지**
2.  에러 발생 시 사용자에게 친절한 UI 알림(예: "데이터를 불러오는 중 문제가 발생했습니다. 새로고침 해주세요") 제공

## 3. 구체적인 실행 계획 (Action Plan)

현재는 코드 수정 없이 계획만 수립하며, 추후 승인이 떨어지면 아래 단계별로 진행합니다.

### Step 1: 강제 로그아웃 로직 전면 제거 (프론트엔드)
*   **대상**: `useAuth` 훅, 전역 에러 핸들러, Firestore 데이터 패칭 컴포넌트
*   **작업**: `permission-denied` 에러를 캐치했을 때 실행되던 `signOut()` 또는 `logout()` 함수 호출 구문을 완전히 삭제합니다.
*   **대체**: 에러 발생 시 상태 값(예: `isError`, `isSyncing`)만 변경하여 화면에 "데이터 동기화 지연" 스낵바나 경고 배지만 띄웁니다.

### Step 2: Firestore 보안 규칙(Security Rules) 완화
*   **대상**: `firestore.rules`
*   **작업**: 
    *   초기 데이터 로딩 시 발생할 수 있는 `null` 참조나 엄격한 필드 검증 규칙을 완화합니다.
    *   인증된 사용자(`request.auth != null`)라면 본인의 소속이나 기본 데이터를 읽어오는 과정에서 튕기지 않도록 `allow read` 조건을 단순화합니다.

### Step 3: Firebase 로컬 캐시(Offline Persistence) 최적화
*   **작업**: 네트워크 오류나 일시적인 토큰 갱신 실패 시, Firestore가 캐시된 데이터를 사용해 우선 화면을 렌더링하도록 강제합니다. 토큰 에러 때문에 흰 화면이나 로그인 창으로 튕기는 것을 방지합니다.

### Step 4: 세션 복구 유틸리티 추가
*   **작업**: 무한 루프 대신, 사용자가 직접 버튼을 눌러 세션을 갱신할 수 있는 "안전한 재연결 버튼"을 제공합니다. (예: "오류가 지속되면 여기를 눌러 재시작하세요")

## 4. 예상되는 결과 및 부작용 검토

| 항목 | 기대 효과 (운영 관점) | 감수해야 할 리스크 (보안 관점) |
| :--- | :--- | :--- |
| **로그아웃 제거** | 더 이상 무한 로그아웃 루프가 발생하지 않음 | 비정상적인 토큰을 가진 기기에서도 화면 프레임(UI)은 보일 수 있음 (데이터는 안보이더라도) |
| **규칙 완화** | 까다로운 권한 조건 때문에 발생하던 억울한 접근 거부(403) 감소 | 의도치 않은 데이터 노출 가능성이 미세하게 증가하나, `request.auth.uid` 기반의 기본 격리만 유지하면 치명적이지 않음 |
| **캐시 우선** | 오프라인이나 네트워크 불안정 환경에서도 앱이 꺼지지 않고 동작 | 사용자가 잠시 동안 과거의 데이터(업데이트 전 데이터)를 볼 가능성 존재 |
