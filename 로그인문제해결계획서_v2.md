## 1. 현재 상황 및 문제점 요약

최근 차량운행일지 서비스에서 지속적으로 발생하고 있는 구글 로그인(Firebase Auth) 관련 이슈들은 사용자가 서비스에 아예 접근하지 못하게 만드는 치명적인 문제를 야기하고 있습니다. 주요 원인은 다음과 같습니다.

- **엄격한 보안 정책의 역효과**: Firestore 데이터 접근 시 권한 오류(Permission Denied, 403)가 발생하면 보안을 위해 사용자를 **강제로 로그아웃**시키는 로직이 작동했습니다.
- **세션과 토큰의 불일치**: 모바일 브라우저나 PWA 환경(특히 인앱 브라우저 등)에서 백그라운드 전환 시 토큰이 만료되거나 Firebase App Check 검증에 실패하는 경우가 빈번합니다.
- **무한 로그아웃 루프**: 토큰은 만료되었으나 브라우저 세션은 남아있는 상태에서, 데이터를 요청하면 403 오류가 발생하고 → 로그아웃 처리 → 다시 자동 로그인 시도 → 토큰 오류 → 403 발생이 반복되는 **무한 루프**에 빠졌습니다.

## 2. 해결 방향: 보안 타협을 통한 '운영 안정성' 확보

**"이제는 보안 문제를 포기할 것은 포기하고 운영의 방향으로 진행한다"**는 방침에 따라, 시스템의 우선순위를 재조정합니다.
사용자가 데이터를 조금 늦게 보거나 일부 검증이 약화되더라도, **서비스 접속 자체가 차단되는 일은 없어야 합니다.**

### 🛑 포기/완화할 보안 정책
1. 데이터 접근 실패 시 즉각적인 세션 파기(강제 로그아웃) 중단
2. 지나치게 엄격한 Firestore 보안 규칙 완화 (단, **무차별 완화는 금지** — §4-2 참조)
3. App Check 등의 환경 검증 실패 시 하드 블로킹 완화

### 🟢 강화할 운영 정책
1. 어떤 에러가 발생해도 세션(로그인 상태)은 유지 — **단, 진짜 만료(`unauthenticated`)는 예외** (§4-3 참조)
2. 에러 발생 시 사용자에게 친절한 UI 알림(예: "데이터를 불러오는 중 문제가 발생했습니다. 새로고침 해주세요") 제공

## 4. 구체적인 실행 계획 (Action Plan v2)

### Step 0: App Check 강제 정책(Enforcement) 상태 확인 — **0순위, 코드 변경 없음**

가장 빠른 운영 안정화 수단입니다. Firebase Console → App Check → Firestore/Functions의 **Enforce가 켜져 있다면 Unenforced로 전환**하면, App Check 토큰 발급 실패가 그대로 403으로 이어지는 경로가 차단됩니다.
- 작업: 콘솔 설정 확인 → Unenforced 전환(필요 시)
- 효과: 단 1줄도 안 고치고 모바일/인앱 브라우저의 App Check 실패형 차단 즉시 해소
- 리스크: 봇·스크래핑 대응이 약화되나, 현 단계 서비스 규모에서는 수용 가능

### Step 1: 강제 로그아웃 로직 — **잔여 경로 정리**

- **대상**: `useAuth` 외 **개별 데이터 페이지의 페칭 훅** (운행일지, 예약, 차량, 하이패스 등)
- **작업**:
  - 각 페이지 단의 `onSnapshot` / `getDocs` 에러 핸들러에서 `permission-denied` 발생 시 세션을 건드리지 않도록 일관성 점검
  - 직접 캐치 경로에서 Sentry로 흘러가는 `permission-denied` 이벤트도 필터링 (현재 `unhandledrejection`만 차단됨)
- **대체 동작**: 상태 값(`isError`, `isSyncing`)만 변경 → 화면에 "데이터 동기화 지연" 스낵바/배지

### Step 2: Firestore 보안 규칙 — **단순 완화 대신 "Claims Fallback" 도입**

**v1의 위험 분석 누락**: 현재 룰은 전부 `request.auth.token.orgId`(Custom Claims) 기반입니다. 이를 단순히 `request.auth != null`로 낮추면 **다른 기관 데이터까지 노출**되는 격리 붕괴가 발생합니다. 실제 원인은 룰이 엄격해서가 아니라 **토큰의 claims가 비어있거나 stale한 순간**이 있다는 점입니다.

- **대상**: [firestore.rules](firestore.rules) 의 `userOrgId()` / `isOrgMember()` 등 헬퍼
- **작업**: claims가 비어있을 때만 **한 단계 fallback**으로 user 문서를 조회하도록 변경:
  ```
  function userOrgId() {
    return request.auth.token.orgId != null
      ? request.auth.token.orgId
      : get(/databases/$(database)/documents/users/$(request.auth.uid)).data.organizationId;
  }
  ```
- **트레이드오프**: 룰 평가당 `get()` 1회 비용 증가(대개 캐시됨). 격리는 유지.
- **금지 사항**: `allow read: if request.auth != null` 같은 무차별 완화는 **하지 않음**.

### Step 3: 캐시 우선 렌더링 — **이미 활성, 추가는 UX 신호**

`persistentLocalCache` 25MB는 이미 동작 중이므로 추가할 것은 **사용자에게 캐시 상태를 알리는 신호**입니다.
- `onSnapshot` 콜백에서 `snapshot.metadata.fromCache`를 읽어 "오프라인 데이터 표시 중" 배지를 표시
- 권한 에러 발생 시 `getDocFromCache`로 stale 데이터라도 우선 표시 후, 백그라운드 재시도

### Step 4: 세션 복구 유틸리티 — **구체화**

v1의 "안전한 재연결 버튼"을 동작 수준까지 정의합니다.
- **트리거 조건**: `permission-denied` 재시도 2회 모두 실패한 시점
- **노출 위치**: 전역 토스트의 액션 버튼 + 대시보드 상단 경고 배지
- **동작**: `auth.signOut()` → `sessionStorage.clear()` → `window.location.href = '/login'`
- **카피**: "오류가 지속되면 여기를 눌러 다시 로그인하세요"

### Step 5: `unauthenticated`는 로그아웃 유지 — **좀비 세션 재발 방지**

"모든 권한 오류 = 세션 유지"로 묶으면 토큰 revoke / 사용자 삭제 같은 **진짜 만료 케이스**에서 좀비 세션이 다시 생깁니다.
- `permission-denied` → 세션 유지 + 토스트 (현행 유지)
- `unauthenticated` / `auth/user-disabled` / `auth/user-token-expired` 중 refresh 불가 → **명시적 로그아웃 + 로그인 화면**
- 구분 로직은 `useAuth`의 onSnapshot 에러 핸들러와 `refreshToken` catch 블록에 분기 추가

### Step 6: 신규 가입 Race Condition 처리

[useAuth.tsx:166-173](src/hooks/useAuth.tsx#L166-L173)의 "사용자 문서가 없거나 삭제됨" 분기는 **Cloud Function이 user doc을 만들기 전에 onSnapshot이 먼저 발화**할 때도 진입합니다. 룰 완화로는 해결 불가.
- 옵션 A: 클라이언트에서 user doc 부재 시 N초간 대기 후 재구독 (현재 일부 적용된 `isNewlyCreated` 로직을 user doc 레벨로 확장)
- 옵션 B: Cloud Function을 트랜잭션으로 변경해 `users/{uid}` 생성을 회원가입 흐름의 동기 단계로 이동

## 5. 성공 지표 및 롤백 기준

### 5.1 성공 지표 (배포 후 1~2주 모니터링)
- Sentry `permission-denied` 발생률: 기준 대비 **80% 이상 감소**
- 일일 강제 로그아웃 이벤트(`auth/sign-out` 자동 트리거) 수: **0건 수렴**
- 사용자 문의(로그인/접속 불가) 티켓 수: 기준 대비 **50% 이상 감소**

### 5.2 롤백 기준
- 의도치 않은 타 기관 데이터 노출이 1건이라도 확인되면 **Step 2(룰 fallback)는 즉시 롤백**
- 좀비 세션이 다시 보고되면 **Step 5의 분기 조건을 재검토**
- App Check Unenforced 전환 후 비정상 트래픽(봇/스크래핑)이 측정되면 단계적 재활성화

## 6. 예상되는 결과 및 부작용 검토

| 항목 | 기대 효과 (운영 관점) | 감수해야 할 리스크 (보안 관점) |
| :--- | :--- | :--- |
| **App Check Unenforced** | 모바일/인앱 브라우저 차단 즉시 해소 | 봇·스크래핑 방어력 저하 |

| **`unauthenticated` 분기 로그아웃** | 좀비 세션 재발 방지 | 정상 만료 사용자는 다시 로그인 필요 (정상 동작) |
