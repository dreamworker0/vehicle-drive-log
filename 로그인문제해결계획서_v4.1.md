## 4. 구체적인 실행 계획 (Action Plan v4.1)

### Step 0: App Check Enforcement 완화 — 즉시 (콘솔)

Firebase Console → App Check → Firestore/Functions의 Enforce가 켜져 있다면 **Unenforced로 전환**. 안정화 후 ReCaptcha Score Threshold 재조정은 후속 과제.

### Step 1: 잔여 경로 정리 + **Sentry 분류 태그 도입** (작업 시간 ~1시간)

#### 1-A. 4개 파일 일관성 점검

- [src/hooks/useVehicleManager.ts](src/hooks/useVehicleManager.ts)
- [src/hooks/useTodayDashboard.ts](src/hooks/useTodayDashboard.ts)
- [src/components/admin/AdminDashboard.tsx](src/components/admin/AdminDashboard.tsx)
- [src/components/superAdmin/SuperAdminLayout.tsx](src/components/superAdmin/SuperAdminLayout.tsx)

확인: `permission-denied` catch 블록 내 `signOut`/`auth` 직접 호출 없는지. 대체 동작은 상태값(`isError`, `isSyncing`)만 변경 + "데이터 동기화 지연" 스낵바.

#### 1-B. Sentry 분류 태그 — **Step 2 모니터링의 전제 조건**

태그 없이는 1~2주 후 모인 `permission-denied` 이벤트가 어느 경로에서 왔는지 분류 불가 → Step 4/5의 진행 여부를 판단할 수 없음. **반드시 Step 1과 함께 도입**.

**적용 위치**: 4개 페이지 훅 + [useAuth.tsx:176-198](src/hooks/useAuth.tsx#L176-L198) (userWatch 에러), [useAuth.tsx:105-119](src/hooks/useAuth.tsx#L105-L119) (orgWatch 에러).

**필수 태그/Extra**:
```typescript
Sentry.captureException(error, {
  tags: {
    source: 'useTodayDashboard' | 'useVehicleManager'
          | 'AdminDashboard' | 'SuperAdminLayout'
          | 'useAuth.userWatch' | 'useAuth.orgWatch',
    phase: 'initial-load' | 'visibility-resume' | 'snapshot-error',
    errorCode: error.code,        // 'permission-denied' 등
  },
  extra: {
    isNewUser: createdMillis > 0 && (Date.now() - createdMillis) < 60_000,
    retryCount,                    // 재시도 횟수 (useAuth만)
    hasOrgIdClaim: !!claims?.orgId,
  },
});
```

**주의**: `permission-denied`는 이미 [firebase.ts:171-176](src/lib/firebase.ts#L171-L176)의 `unhandledrejection` 핸들러에서 Sentry 전송을 억제 중입니다. 명시적 `captureException`만 통과되므로 위 태그가 그대로 Sentry에 적재됩니다.

### Step 2: Sentry 모니터링 (1~2주) — 데이터 기반 후속 결정

Step 0/1 적용 후 다음 분류로 집계:

- `source` × `errorCode`별 발생률 → Step 1에서 빠뜨린 경로 식별
- `isNewUser=true` 비율 → Step 4 진행 여부 결정 (주 1회 이상 발생 시 진행)
- `useAuth.userWatch` + `retryCount=2` 잔존률 → Step 5(룰 fallback) 진행 여부 결정
- `phase=visibility-resume` 비율 → 백그라운드 복귀 토큰 갱신 추가 보강 필요성 확인

### Step 3: 재연결 토스트 버튼 + 진행 상태 표시

#### 3-A. "다시 로그인" 버튼 — 5단계 정리 절차

[useAuth.tsx:190](src/hooks/useAuth.tsx#L190) 토스트에 버튼 추가. 클릭 시 다음 순서로 정리(IDB 캐시 꼬임 이력 `9bcb055` 반영):

1. `await auth.signOut().catch(() => {})`
2. `await clearIndexedDbPersistence(db).catch(() => {})` — 유틸은 [firebase.ts:122-144](src/lib/firebase.ts#L122-L144)에 이미 존재
3. **선택적 localStorage 키만 삭제** — `localStorage.clear()` 금지(아래 §3-B 근거)
   - 삭제: `pendingInviteCode`
   - 보존: `theme-preference`, `font-size`, `preferred-nav-app`, `employee-welcome-dismissed`, `install-*-dismissed` 등 UX 설정
4. `sessionStorage.clear()`
5. `window.location.replace('/login')` — `href` 대신 `replace`로 히스토리 정리

#### 3-B. localStorage 전체 삭제 금지 — 실측 근거

grep 결과 앱이 localStorage에 저장하는 키는 다음 13종이며, 대부분 **UX 설정으로 인증 문제와 무관**합니다.

| 키 | 용도 | 재연결 시 처리 |
|---|---|---|
| `pendingInviteCode` | 가입 플로우 임시 상태 | **삭제** |
| `theme-preference`, font-size, `preferred-nav-app` | 사용자 설정 | 보존 |
| `employee-welcome-dismissed`, `install-*-dismissed`, `driveLog_*Expanded`, `admin-onboarding-*` | 1회성 안내 dismiss 상태 | 보존 |
| `sa-test-role` (`SA_TEST_ROLE_KEY`) | superAdmin 임시 역할 전환 | 보존 (테스트 도구) |
| tmap geo/route 캐시 | API 비용 절감용 | 보존 |

`localStorage.clear()`는 위 모든 UX 설정을 날려 사용자 경험을 손상시키므로 **선택적 삭제만 허용**.

#### 3-C. 재시도 중 진행 상태 표시 — UX 보강

실제 사용자가 멈춰 보이는 구간은 [useAuth.tsx:176-186](src/hooks/useAuth.tsx#L176-L186)의 userWatch 재시도 ~3초(`loading=true` 유지). 이 동안 화면에 다음 중 하나가 보이도록 점검:

- 전역 로딩 스피너가 이미 떠 있는지 (`loading` state 소비처 확인)
- 2초 이상 지속 시 "권한 정보를 동기화 중입니다…" 보조 텍스트 노출

### Step 4 (조건부): 신규가입 race condition 완화

**조건**: Step 2 모니터링에서 `extra.isNewUser=true` 이벤트가 주 1회 이상.

- **1차 시도 (1줄 변경)**: [useAuth.tsx:131](src/hooks/useAuth.tsx#L131)의 `setTimeout(startOrgWatch, 500)`을 **1500~2000ms로 확대**.
- **스피너 추가 권고는 기각**: 외부 의견은 "2초 화면 정지"를 우려했으나, 해당 `setTimeout`은 `startOrgWatch`만 지연시키며 [useAuth.tsx:136](src/hooks/useAuth.tsx#L136)의 `setLoading(false)`를 블로킹하지 않습니다. UI는 멈추지 않고 단지 `orgDeleted` 감지가 늦어질 뿐 — 신규가입 직후엔 시각적 차이가 없습니다.
- **진짜 멈춤 구간은 Step 3-C에서 다룹니다.**
- **2차 시도(여전히 발생 시)**: users 문서 폴링 스플래시 도입.

### Step 5 (조건부): Firestore 룰 Claims Fallback

**조건**: Step 2 모니터링에서 `source=useAuth.userWatch` + `retryCount=2` + `hasOrgIdClaim=false` 이벤트가 지속.

```javascript
function userOrgId() {
  return request.auth.token.orgId != null
    ? request.auth.token.orgId
    : get(/databases/$(database)/documents/users/$(request.auth.uid)).data.organizationId;
}
```

**비용 주의**: 룰 내 `get()`은 매 read마다 1회 추가 과금. 리스트 쿼리에는 적용되지 않아 효과 제한적. 데이터로 입증된 경우에만 진행.

## 5. 성공 지표 및 롤백 기준

### 5.1 성공 지표 (1~2주)
- Sentry `permission-denied` 발생률: 기준 대비 **80% 이상 감소**
- 일일 자동 강제 로그아웃 이벤트: **0건 수렴**
- 사용자 문의 티켓: 기준 대비 **50% 이상 감소**
- **태그 기반 신규 지표**: `isNewUser=true` 건수 / `phase=visibility-resume` 건수 / `useAuth.userWatch.retryCount=2` 잔존률

### 5.2 롤백 기준
- App Check Unenforced: 비정상 트래픽/할당량 폭증 시 즉시 Enforced 복귀
- Step 5 룰 fallback: 타 기관 데이터 노출 1건 확인 시 **즉시 롤백**
- Billing 비정상 상승 시 Step 5 재검토

## 6. 작업 순서 요약 (v4.1)

1. **(0순위, 즉시)** App Check Enforcement 콘솔 점검 → Unenforced 전환
2. **(1순위, ~1시간)** Step 1-A (4개 파일 점검) + **Step 1-B (Sentry 태그 도입)**
3. **(1순위)** Step 3-A (재연결 토스트 버튼, 5단계 정리) + 3-C (진행 상태 표시 점검)
4. **1~2주 Sentry 지표 / Billing 모니터링** (Step 2 — 태그 기반 분류)
5. **데이터 기반 후속 결정**:
   - `isNewUser` 비율 높음 → Step 4 (1줄 지연 확대 → 필요 시 폴링)
   - `userWatch retryCount=2` 잔존 → Step 5 (룰 fallback)
6. 안정화 후 App Check Score Threshold 재조정
