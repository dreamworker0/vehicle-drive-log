## 1. 현장 로그가 드러낸 추가 위협

2026-05-12 00:49~00:51 콘솔 로그에서 다음이 확인됨:

```
content-firebaseappcheck.googleapis.com/.../exchangeRecaptchaV3Token: 500
@firebase/app-check: AppCheck: 500 error. Attempts allowed again after 00m:01s (appCheck/initial-throttle).
@firebase/auth: Auth (12.9.0): Error while retrieving App Check token: FirebaseError: AppCheck: 500 error.
```

### 인과 체인
1. Google App Check 서버(`content-firebaseappcheck.googleapis.com`)가 5xx 반환 (Google 인프라 장애 또는 사이트 키 설정 문제)
2. SDK가 지수 백오프 throttle 진입 (1s → 2s → 4s → 8s → 11s)
3. **Auth + Firestore가 App Check 토큰을 못 받음** → 요청에 토큰 없이 전송
4. **Enforcement=ON**이면 백엔드가 거부 → `permission-denied` → useAuth가 재시도/세션 처리 → 무한 루프
5. 동시에 console.warn이 매 토큰 실패마다 1줄씩 출력 → 1분 내 100줄+ 노이즈

### 핵심 인식
- v4.1까지의 "로그아웃 루프 차단"으로는 **백엔드 거부 자체를 막을 수 없음** → Step 0의 Unenforced 전환이 진짜 출혈 방지선
- 다만 Unenforced여도 **throttle 백오프로 인한 첫 로딩 지연**과 **콘솔 노이즈**는 별도 처리 필요

## 2. 구체적인 실행 계획 (Action Plan v4.2)

### Step 0: App Check Enforcement 완화 — 즉시 (콘솔)

Firebase Console → App Check → 각 서비스별 Enforce 상태 점검 후 **모두 Unenforced로 전환**:

- **Firestore** (Unenforced)
- **Cloud Functions** (Unenforced)
- **Identity Toolkit (Authentication)** ← v4.2에서 추가 명시. 로그의 `@firebase/auth: Error while retrieving App Check token`이 발생하는 경로 차단
- **Storage** (사용 중이면 Unenforced)

### Step 0-B: reCAPTCHA 설정 점검 (신설, 즉시)

500이 **단기 인프라 장애가 아닌 설정 문제**일 가능성을 즉시 배제해야 합니다.

**체크리스트**:
1. **Firebase Console → App Check → 앱 등록**
   - 현재 운영 도메인이 reCAPTCHA v3 공급자로 등록되어 있는가?
   - 사이트 키가 콘솔에 등록된 것과 빌드 시 주입되는 `VITE_RECAPTCHA_SITE_KEY`와 일치하는가?
2. **Google reCAPTCHA Admin Console** (https://www.google.com/recaptcha/admin)
   - 사이트 키의 도메인 화이트리스트에 운영 도메인이 포함되어 있는가?
   - 키 타입이 reCAPTCHA v3인가? (v2/Enterprise 혼동 주의)
3. **빌드 시점 환경변수 확인**
   - `.env.production` 또는 배포 환경의 `VITE_RECAPTCHA_SITE_KEY`가 **개발 키와 섞이지 않았는가**
   - [firebase.ts:36](src/lib/firebase.ts#L36)의 `import.meta.env.VITE_RECAPTCHA_SITE_KEY`가 빈 문자열이 아닌지 빌드 산출물에서 확인
4. **iframe/CSP 점검**
   - reCAPTCHA가 `www.google.com/recaptcha`에 iframe을 띄움 — CSP `frame-src`가 차단하지 않는지 확인
   - PWA standalone 모드에서 reCAPTCHA가 정상 동작하는지 모바일 실기기 확인

→ 위 4개 중 하나라도 미설정/불일치면 500 폭주의 직접 원인일 수 있으므로 **모니터링보다 우선 조치**.

#### 1-C. **App Check 콘솔 노이즈 디둡 (v4.2 신설)**

[firebase.ts:41-51](src/lib/firebase.ts#L41-L51)의 `onTokenChanged` error 핸들러는 매 토큰 실패마다 `console.warn` 1줄을 출력 → 1분 내 100줄+ 발생 사례 확인됨.

**적용**: 동일 `errorCode` 기준 60초 윈도우에 1회만 출력하도록 디둡.

```typescript
const appCheckWarnDedup = new Map<string, number>();
onTokenChanged(appCheck, {
  next: () => {},
  error: (err) => {
    const code = (err as { code?: string })?.code || 'unknown';
    const now = Date.now();
    const last = appCheckWarnDedup.get(code) ?? 0;
    if (now - last < 60_000) return;       // 60초 윈도우
    appCheckWarnDedup.set(code, now);
    const isLocalhost = window.location.hostname === 'localhost'
                      || window.location.hostname === '127.0.0.1';
    console.warn(
      `[App Check] 토큰 발급 실패 (${code}) — ${isLocalhost ? '로컬' : '인프라/네트워크'} 이슈로 간주:`,
      err
    );
  }
});
```

추가로 **누적 카운터**를 두어 60초마다 "지난 1분간 N회 발생" 형태로 한 줄 요약 출력하면 운영자 디버깅에 유용. (선택)

### Step 3: 재연결 토스트 + 진행 상태 표시 (v4.1 기반, 3-C 확장)

#### 3-C. **재시도/throttle 중 진행 상태 표시 (v4.2 확장)**

**구현 권고**:
