## 🟢 오전에 해도 되는 일 (Risk: 🔴 낮음)
실제 운영 서버(Firebase)와 사용자 환경에 영향을 주지 않는 순수 로컬/설정 준비 작업입니다. 언제든 편하게 진행하셔도 무방합니다.

### 1️⃣ GitHub Personal Access Token (PAT) 발급 및 갱신
* **내용**: GitHub 설정에서 `workflow` 권한이 포함된 새 토큰을 발급받고 로컬 PC의 Git 자격증명(Credential)을 업데이트합니다.
* **이유**: 운영 서버를 건드리지 않으며, 저녁 7시에 진행할 강제 푸시를 위한 단순 준비 작업입니다.

### 2️⃣ 추가 기능 로컬 개발 및 테스트
* **내용**: 대시보드 지표 개선이나 예약 기능 추가 등 새로운 코드를 짜고 `npm run dev`나 `npm test`를 돌려보는 일.
* **이유**: 전부 작성자 컴퓨터(PC) 환경에서만 돌아가므로, 현업 사용자들이 쓰는 실제 화면이나 데이터베이스에는 아무 영향이 없습니다.

### 3️⃣ (권장) Firebase 배포를 대비한 롤백(Rollback) 대책 시뮬레이션
* **내용**: 만약 저녁 7시 배포 후 "화면이 하얗게 뜬다" 거나 함수 에러가 발생할 때, 즉시 이전 상태의 Hosting과 Functions 버전을 복구할 수 있는 방법을 메모해 둡니다.

## 🔴 저녁 7시 이후에 해야 하는 일 (Risk: 🔴 매우 높음)
사용자들의 데이터 변경이나 화면 갱신에 직접적인 영향을 주며, 짧게는 1분에서 길게는 10분 정도의 서비스 지연/오작동을 발생시킬 수 있는 작업들입니다. **사용량이 급감하는 19시 이후로 미루세요.**

### 1️⃣ GitHub 원격 저장소 강제 덮어쓰기 (`git push -f`)
* **이유**: GitHub 저장소에 코드가 덮어씌워지며 변경된 CI/CD 파이프라인이 즉각적으로 작동할 수 있습니다. 기존에 배포(Deploy) 자동화가 걸려 있다면 낮에 쏘는 순간 배포가 시작되므로 주의가 필요합니다.

### 2️⃣ Firebase Admin SDK 키(JSON) 폐기 및 재발급
* **이유**: [가장 핵심적인 이유] 보안을 위해 Firebase 콘솔에서 **기존 서비스 계정(열쇠)을 지우는 순간, 그 계정에 의존해서 구동되고 있던 Cloud Functions나 일부 서버의 호출 권한이 막혀 에러(403 Forbidden 등)가 발생합니다.** 새 키를 발급받아 환경변수에 이식하기까지 생기는 공백 시간 동안 사용자는 에러를 경험하게 됩니다.

### 3️⃣ Firebase Cloud Functions 운영 서버 배포
* **이유**: 대대적인 로직 변경(타임아웃, 메모리 상승, 서브모듈 분리 등)이 있었기 때문에 배포 과정 자체에서 기존 인스턴스가 내려가고 스케일되는 동안 연결된 사용자들의 저장/예약 승인 버튼이 "응답 없음"이 될 수 있습니다.

### 4️⃣ 프론트엔드 React 운영 앱 배포 (Hosting)
* **이유**: 컴포넌트 덩어리가 완전히 분리되어 번들(Bundle)이 새롭게 만들어졌습니다. 업무 시간에 배포할 경우 기존 접속자들이 앱을 작동 중에 **"청크(Chunk) 파일을 찾을 수 없습니다"**라는 에러와 함께 화면 단절 현상(하얀 화면 모바일 화이트 스크린)을 매우 높은 확률로 겪게 됩니다.

### 5️⃣ 예약 관리자 승인 기능 배포 및 실제 테스트 (Pending 상태 적용)
* **내용 및 이유**: `createReservationSafe` 함수(Cloud Functions)가 새 배포판으로 갱신되어야 클라이언트의 `pending`(승인 대기) 상태를 서버에서 정상적으로 접수합니다. 3번의 백엔드 배포와 동반되며, 배포 직후 일반 사용자 계정으로 예약을 넣어 '승인 대기' 빗금 UI가 노출되는지 실테스트를 진행합니다.

> [!TIP]

> 1. 오전에 준비해 둔 GitHub 새 토큰으로 `git push origin master --force` 실행
> 2. Firebase 접속 -> 수동으로 기존 키 지우고 새로운 `vehicle-drive-log-firebase-adminsdk-*.json` 다운로드 (저녁 7시 시작의 신호탄)

> 4. 최종적으로 프론트엔드 `firebase deploy --only hosting` 배포
> 5. 모바일/웹 접속 후 정상 렌더링 확인

### 🚨 긴급 배포 롤백 (Rollback) 시나리오

1. **프론트엔드 (화면 백화현상 등) 오류 시:** 
   - [Firebase Console -> Hosting] 접속
   - '출시 내역'에서 직전 정상 버전의 우측 점 3개(⋮) 클릭 -> **[롤백]** 누르기 (즉시 복구됨)

2. **백엔드 (Functions) 치명적 오류 파악 시:**
   - 로컬 터미널에서 즉시 이전 커밋으로 되돌림: `git revert HEAD`
   - Functions만 긴급 재배포: `firebase deploy --only functions`
   *(재배포 속도가 너무 느리고 1분 1초가 급할 때는 Google Cloud Console -> 'Cloud Run'의 해당 함수 '리비전' 탭에서 트래픽을 이전 리비전으로 100% 돌리면 즉시 복구)*
