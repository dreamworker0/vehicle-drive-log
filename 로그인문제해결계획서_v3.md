## 1. 현재 상황 및 문제점 요약

최근 차량운행일지 서비스에서 지속적으로 발생하고 있는 구글 로그인(Firebase Auth) 관련 이슈들은 사용자가 서비스에 아예 접근하지 못하게 만드는 치명적인 문제를 야기하고 있습니다. 주요 원인은 다음과 같습니다.

- **엄격한 보안 정책의 역효과**: Firestore 데이터 접근 시 권한 오류(Permission Denied, 403)가 발생하면 보안을 위해 사용자를 **강제로 로그아웃**시키는 로직이 작동했습니다.
- **세션과 토큰의 불일치**: 모바일 브라우저나 PWA 환경(특히 인앱 브라우저 등)에서 백그라운드 전환 시 토큰이 만료되거나 Firebase App Check 검증에 실패하는 경우가 빈번합니다.
- **무한 로그아웃 루프**: 토큰은 만료되었으나 브라우저 세션은 남아있는 상태에서, 데이터를 요청하면 403 오류가 발생하고 → 로그아웃 처리 → 다시 자동 로그인 시도 → 토큰 오류 → 403 발생이 반복되는 **무한 루프**에 빠졌습니다.

## 2. 해결 방향: 보안 타협을 통한 '운영 안정성' 확보

**"이제는 보안 문제를 포기할 것은 포기하고 운영의 방향으로 진행한다"**는 방침에 따라, 시스템의 우선순위를 재조정합니다.
사용자가 데이터를 조금 늦게 보거나 일부 검증이 약화되더라도, **서비스 접속 자체가 차단되는 일은 없어야 합니다.**

### 🛑 포기/완화할 보안 정책
1. 데이터 접근 실패 시 즉각적인 세션 파기(강제 로그아웃) 중단
2. 지나치게 엄격한 Firestore 보안 규칙 완화 (단, **무차별 완화는 금지** — §4-2 참조)
3. App Check 등의 환경 검증 실패 시 하드 블로킹 완화

### 🟢 강화할 운영 정책
1. 어떤 에러가 발생해도 세션(로그인 상태)은 유지 — **단, 진짜 만료(`unauthenticated`)는 예외** (§4-3 참조)
2. 에러 발생 시 사용자에게 친절한 UI 알림(예: "데이터를 불러오는 중 문제가 발생했습니다. 새로고침 해주세요") 제공

## 4. 구체적인 실행 계획 (Action Plan v3)

### Step 0: App Check 강제 정책(Enforcement) 완화 및 장기 대안 모색

가장 빠른 운영 안정화 수단입니다. Firebase Console → App Check → Firestore/Functions의 **Enforce가 켜져 있다면 Unenforced로 전환**하면, App Check 발급 실패가 바로 차단으로 이어지지 않습니다.
- **즉시 조치**: 콘솔 설정 확인 → Unenforced 전환
- **장기 대안 (안정화 이후)**: Unenforced 방치 시 할당량 공격 위험이 있으므로, ReCaptcha Enterprise v3의 검증 임계값(Score Threshold)을 완화하거나 모바일/인앱 브라우저의 `initializeAppCheck` 로직 지연 여부를 점검하는 후속 작업을 계획합니다.

### Step 1: 강제 로그아웃 로직 — **잔여 경로 정리**

- **대상**: `useAuth` 외 **개별 데이터 페이지의 페칭 훅** (운행일지, 예약, 차량, 하이패스 등)
- **작업**:
  - 각 페이지 단의 `onSnapshot` / `getDocs` 에러 핸들러에서 `permission-denied` 발생 시 세션을 건드리지 않도록 일관성 점검
  - 직접 캐치 경로에서 Sentry로 흘러가는 `permission-denied` 이벤트도 필터링
- **대체 동작**: 상태 값(`isError`, `isSyncing`)만 변경 → 화면에 "데이터 동기화 지연" 스낵바/배지 표시

### Step 2: Firestore 보안 규칙 — **"Claims Fallback"과 비용 방어**

현재 룰은 `request.auth.token.orgId` 기반입니다. 토큰이 stale할 때를 대비해 한 단계 fallback으로 user 문서를 조회합니다. 단, 무차별 완화(`request.auth != null`)는 격리를 붕괴시키므로 절대 금지합니다.
- **룰 수정**:
  ```javascript
  function userOrgId() {
    return request.auth.token.orgId != null
      ? request.auth.token.orgId
      : get(/databases/$(database)/documents/users/$(request.auth.uid)).data.organizationId;
  }
  ```
- **비용 방어 로직 (클라이언트)**: 
  - 보안 규칙 내 `get()` 호출은 **실제 1회의 읽기 과금**을 발생시킵니다.
  - 리스트 조회 시 수십 번의 `get()` 호출 폭탄을 막기 위해, 클라이언트 로그인 직후 `IdTokenResult`의 claims(`orgId`)가 비어 있다면 **데이터 페칭 전 `auth.currentUser.getIdToken(true)`를 의도적으로 강제 호출**하여 토큰을 최신화하는 방어막을 `useAuth`에 추가합니다.

### Step 3: 세션 복구 및 `unauthenticated` 완벽 분리

"모든 권한 오류 = 세션 유지"로 묶으면 사용자가 진짜 만료(삭제 등)되었을 때 좀비 세션이 부활합니다.
- **분기 처리 로직 (`onIdTokenChanged` 활용)**:
  - `firebase/firestore`의 `permission-denied`: 세션 유지 + 오프라인/재시도 상태 알림 (현행 유지)
  - `auth` 모듈의 만료/삭제 에러 또는 `onIdTokenChanged`에서 User 객체가 null이 되는 시점(진짜 만료): **명시적 로그아웃 + 로그인 화면 리다이렉트**
- **재연결 유틸리티**: `permission-denied` 2회 실패 시 전역 토스트에 "오류가 지속되면 여기를 눌러 다시 로그인하세요" 버튼 노출 (`sessionStorage.clear()` 및 강제 리로드 포함).

### Step 4: 신규 가입 Race Condition 처리 (로딩 폴링 로직)

가입 직후 Cloud Function이 `users` 문서를 만들기 전에 프론트가 데이터를 요청하여 발생하는 403 에러를 방지합니다.
- **안전한 클라이언트 제어(Polling)**: "회원가입 직후(초기화 단계)라면, `users` 문서가 존재하는지 클라이언트에서 짧은 주기로 폴링하여 확인한 후, 문서가 확인되었을 때 메인 화면으로 진입"하도록 하는 **로딩/스플래시 화면**을 도입하여 엣지 케이스를 원천 차단합니다.

## 5. 성공 지표 및 롤백 기준

### 5.1 성공 지표 (배포 후 1~2주 모니터링)
- Sentry `permission-denied` 발생률: 기준 대비 **80% 이상 감소**
- 일일 강제 로그아웃 이벤트(`auth/sign-out` 자동 트리거) 수: **0건 수렴**
- 사용자 문의(로그인/접속 불가) 티켓 수: 기준 대비 **50% 이상 감소**

### 5.2 롤백 기준
- 의도치 않은 타 기관 데이터 노출이 1건이라도 확인되면 **Step 2(룰 fallback)는 즉시 롤백**
- Firestore 읽기 비용(Billing)이 비정상적으로 치솟는 경우 (클라이언트 방어막 누수 시)
- 좀비 세션이 다시 보고되면 **Step 3의 분기 조건을 재검토**
